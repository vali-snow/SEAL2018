Report A

The Choice Between SPA and Browser Extension

Our application will be of type SPA - Single Page Application.

That means we will load a single HTML page and dynamically update it as the user interacts with the App.

This approach allows the creation of a fluid and responsive App, without the constant page reloads.

However, this also has the drawback of a slower initial load.

Modules

We have structured our application in 4 main modules:

⚬ app.js

This is for initializing the Web App, and setting things in motion for loading the default view.

⚬ db.js

Due to the lack of a specific Web API, we will use a mocked database, in the form of a JSON file.

⚬ router.js

This module is the core of our application. Its primary role is switching between our views. And added onto that, it generates the dynamic content of each specific view, based on the information retrieved from the mocked database.

⚬ notification.js

Used to implement the notifications of the Web App.

How It Works - An Eagle's Eye View

Our SPA consists of one index.html, and 4 views: main.html, result.html, allergy.html and report.html.

The default view on which our web application starts is main.html. We set it as default in our app.js module.

Our app.js module creates an instance of router.js. Here, we add an eventListener which awaits page changes, as resulting from using the Web App by clicking on links.

Whenever a link is clicked, the event is triggered, and the div in index.html, the one with the app class, is changed with another div containing the the requested information, simulating a page-change without actually changing the page.

Now, if the page has a parameter, as for example if we request the allergy view, for allergy with id 4, router.js uses the loadData method to fetch the information from our mockedDatabase, select the one with the proper id, dinamically generate part of the replacing div and replace it in index.html
